## Задача

>    Описание проекта
>
>    Разрабатываемый продукт — мессенджер с возможностью обмена сообщениями и создания групповых чатов, аналогичный Telegram.
>
>    Задача: Разработать мини-приложение, реализующее функционал чата с возможностью отправки сообщений, их хранения в базе данных и создания групповых чатов.
>
>    Функциональные требования
>    1. Чат (WebSocket)
>        - Реализовать подключение пользователей через WebSocket.
>        - Обмен текстовыми сообщениями между пользователями в реальном времени.
>        - Возможность создания групповых чатов.
>        - Сообщения должны сохраняться в PostgreSQL.
>        - Каждое сообщение должно содержать:
>        - ID сообщения
>        - ID чата
>        - ID отправителя
>        - Текст
>        - Время отправки
>        - Признак прочитанного сообщения
>        - Реализовать обработку статуса "прочитано" (отправитель получает уведомление, когда сообщение прочитано получателем или всеми участниками группы).
>        - Предотвращение дублирования сообщений при параллельной отправке.
>    2. История сообщений
>        - Реализовать REST-эндпоинт для получения истории сообщений по ID чата:
>        - Метод: GET /history/{chat_id}
>        - Параметры запроса: chat_id — ID чата (обязательный параметр), limit и offset для пагинации (опционально).
>        - Сообщения должны быть отсортированы по времени отправки (по возрастанию).
>
>   Технические требования
>    1. Backend:
>        - FastAPI для реализации REST API и WebSocket.
>        - Асинхронная работа через asyncio.
>        - PostgreSQL как основная база данных.
>        - SQLAlchemy (использование ORM, асинхронные запросы).
>        - Контейнеризация (Docker, Docker Compose).
>        - Автоматическая документация (Swagger / OpenAPI).
>    2. Структура базы данных:
>        - users (id, имя, email, пароль).
>        - chats (id, название, тип: личный/групповой).
>        - groups (id, название, создатель, список участников).
>        - messages (id, chat_id, sender_id, text, timestamp, прочитано).

## Результат

Запуск приложения реализован в контейнере, точнее - сети контейнеров: непосредственно веб-приложение, БД, кэш. Контейнеры объединены в одну сеть внутри Docker и, возможно, предварительно необходимо однократно создать соответствующую сеть командой

```bash
	docker network create --driver bridge shared-network
```

Если не создать указанную сеть, Docker будет использовать стандартную сеть типа bridge, но это может потенциально создать конфликты с другими контейнерами, подключенными к этой сети.

Для запуска необходимо обеспечить присутствие в корневой папке проекта файла `.env` аналогичного по структуре [.env.example](./.env.example), после чего выполнить команду

```bash
	docker compose up [-d]
```

Параметр `-d` является опциональным, применяется для запуска контейнеров в [detached-режиме](https://docs.docker.com/engine/containers/run/#foreground-and-background/).

Миграции БД применяются автоматически при запуске контейнера с сервисом. Для персистентного хранения данных БД между запусками сервиса организован именованный Docker volume, кроме того, посредством volume реализовано хранение созданного и примененного файла миграции. Alembic очень чувствительно реагирует, если в БД существует таблица с именем текущей версии файла миграции, однако файл отсутствует.

Интерактивная схема OpenAPI в формате SwaggerUI с доступными HTTP-эндпоинтами доступна по адресу [http://127.0.0.1:8000/docs/](http://127.0.0.1:8000/docs/), поэтому примеры запросов здесь не приводятся.

Для наполнения базы тестовыми данными после запуска необходимо выполнить команду:

```bash
    docker exec wschat python prefill_test_data.py
```

Скрипт добавляет в базу следующие сущности:

- Пользователи[^1]
1) `{'email': 'user1@mail.com', 'name': 'User1', 'password': 'password123'}`
2) `{'email': 'user2@mail.com', 'name': 'User2', 'password': 'password123'}`
3) `{'email': 'user3@mail.com', 'name': 'User3', 'password': 'password123'}`

- Чаты
1) `{'category': 'PRIVATE', 'name': 'some_public_chat', 'participants': [1, 2]}`
2) `{'category': 'PUBLIC', 'name': 'some_private_chat', 'participants': [1, 2, 3]}`

- Группа
1) `{'chat_id': 2, 'creator': 2, 'name': 'some_public_group', 'participants': [1, 2, 3]}`

### Маршруты

Базовый адрес приложения: [http://127.0.0.1:8000/](http://127.0.0.1:8000/).

Реализованный функционал доступен по следующим маршрутам:

| Название | Метод | Описание |
|----------|-------|----------|
| /create/user           | POST | Регистрация нового пользователя
| /create/chat           | POST | Создание нового чата
| /create/group          | POST | Создание новой группы
|						|	   |
| /history/{chat_id}		| GET  | Получение истории сообщений чата с идентификатором `chat_id`
|						|	   |
| /auth/login 			| POST | Авторизация пользователя
| /auth/refresh 			| POST | Обновление access_token
|						|	   |
| /ws/&lt;chat_id&gt;/?token={JWT} | ws     | Вебсокет чат

Для подключения к чату, в БД должны существовать:
- пользователи, от имени которых осуществляется подключение;
- чат, в котором будет происходить общение;
- (опционально) группа для публичного чата.

При создании чата указывается его видовой признак: `PRIVATE` или `PUBLIC`. Создание группы, для участников которой происходит трансляция всех поступающих в чат сообщений, возможно только на основе публичного чата.

Доступ пользователей в чат, участником которого он не является, запрещен по соображениям приватности.

**В проекте реализована аутентификация и авторизация посредством JWT**[^2], эндпоинты для авторизации и обновления `access_token` приведены  в таблице выше.

Для создания подписи ключа используется алгоритм **HS256**, который является симметричным, т.е. использует один ключ для создания подписи и ее валидации. Сам ключ генерируется в момент запуска контейнера с сервисом и контейнер не покидает в целях безопасности.

Переменная окружения `AUTH__ACCESS_TOKEN_EXPIRE_MINUTES` определяет срок действия `access_token` (измеряется в минутах).

Переменная окружения `AUTH__REFRESH_TOKEN_EXPIRE_DAYS` определяет срок действия `refresh_token` (измеряется в сутках).

Для подключения к бэкенд-приложению необходимо в параметрах запроса передать полученный `access_token`[^3].

Создать необходимые модели вручную, помимо создаваемых с помощью [скрипта](./prefill_test_data.py), авторизоваться и получить токен доступа, обновить токен доступа можно самостоятельно в [SwaggerUI](http://127.0.0.1:8000/docs/).

Протестировать работу чатов можно с помощью простого тестового клиентского приложения - [ws_client.html](./ws_client.html).

Для полноценного тестирования реализованного функционала рекомендуется запускать несколько экземпляров клиентского приложения с помощью простого HTTP-сервера, входящего в стандартную библиотеку Python:

```python
	python -m http.server {номер порта}
```

Клиентское приложение будет доступно в браузере по адресу: `http://localhost:{номер порта}/ws_client.html`

После успешного прохождения авторизации токен доступа будет сохранен и автоматически подставлен в параметры подключения к вебсокету, в случае успешного подключения[^4] будет доступен интерфейс чата.

Автообновление токена доступа также для удобства реализовано в клиенте.

Дополнительно необходимо отметить, что именно для удобства подключения клиентских приложений CORS фактически отключен (разрешены все `Origins`, методы и заголовки).

Значение переменной окружения `DEBUG` отличное от `True` вновь вернет механизм CORS в активный режим и потребует ручной настройки разрешенных клиентских хостов на стороне сервера.

**В приложении реализована возможность подключения нескольких клиентских приложений от имени одного пользователя.**

### Логирование

Необходимо отдельно упомянуть про конфигурирование логирования.

Логи записываются в стандартный вывод и лог-файл, находящийся в директории `logs/`. Эта же директория указана в качестве docker volume, пробрасываемого в корневую папку проекта на хосте, что гарантирует ее сохранность даже в случае выхода из строя контейнера с приложением.

Лог-файл автоматически обновляется через конфигурируемый промежуток времени. За это отвечают переменные окружения `APP__LOGFILE_TTL_MEASURE` и `APP__LOGFILE_TTL`. 

Переменная `APP__LOGFILE_TTL_MEASURE` отвечает за тип интервала обновления. Доступны следующие опции:

| Значение |	Тип интервала    |
|----------|---------------------|
|    'S'   | 		Секунды		 |
|    'M'   |		Минуты		 |
|    'H'   |      	Часы		 |
|    'D'   |		Дни			 |

Переменная `APP__LOGFILE_TTL` принимает целое число и отвечает за непосредственно значение выбранного типа интервала.

Например, при значениях переменных `APP__LOGFILE_TTL_MEASURE=D` и `APP__LOGFILE_TTL=30` срок жизни лог-файла составит 30 дней.

При значениях переменных `APP__LOGFILE_TTL_MEASURE=M` и `APP__LOGFILE_TTL=5` срок жизни лог-файла составит 5 минут.

### Дисклеймер

Поскольку данное задание является **тестовым** и не предполагает выхода в продуктовую среду, при его выполнении опущены некоторые стандартные для индустрии шаги, как, например, реализация обратного прокси на Nginx, ограничение доступа к контейнерам напрямую через закрытие портов с хостом, оптимизация работы БД за пределами ORM и т.д.

По этой же причине код снабжен подробными комментариями[^5], заметками и вставками, которые разъясняют нюансы и детали реализации тех или иных компонентов приложения.

На момент сдачи приложение отвечает критериям выполнения тестового задания. Несмотря на это, предполагается, что перспективными направлениями развития аналогичного сервиса могли бы быть:
- внедрение микросервиса-посредника (как вариант: на основе брокера сообщений) для рассылки сообщений пользователям по мере их подключения к чату. 

Необходимость реализации данного механизма не описана в тестовом задании, в связи с чем подобный функционал в приложении отсутствует;
- реализация многопоточной работы приложения. В данный момент "узким" местом является существование единого менеджера вебсокетов, который отвечает за учет, обработку соединений и работу методов доставки сообщений. Uvicorn способен работать в несколько независимых потоков, однако нативная передача между ними экземпляра класса менеджера вебсокетов невозможна по объективной причине. В качестве инструмента синхронизации может выступать Redis при условии сериализации объекта с помощью `pickle` (с оговорками относительно наличия уязвимости в такой сериализации/десериализации) либо изменение подхода работы приложения на использование независимых процессов, синхронизированных с помощью специальных объектов передачи данных между ними (очереди, пайпы и т.д.). 

Вновь необходимо отметить, что задача реализации _масштабируемого_ приложения-мессенджера в ТЗ не ставилась, в связи с чем необходимые подходы своего воплощения в коде не нашли.
